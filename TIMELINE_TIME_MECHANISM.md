# Timeline Time Mechanism

## 목적

이 문서는 `src/App.jsx`의 타임라인 시간 계산 규칙을 사람이 검토 가능한 형태로 고정하기 위한 기준 문서다.

핵심 목표는 아래 3가지다.

1. 모든 일정 시간은 끊기지 않는 연속 타임라인으로 계산한다.
2. `시작시간 고정`, `소요시간 잠금`, `종료시간 잠금`이 있어도 계산을 포기하지 않는다.
3. 일반 계산으로 안 맞는 차이는 `이동칩의 보정시간(buffer)`으로 흡수한다.

---

## 기본 용어

- `이전 일정 종료`: 이전 일정의 `시작시간 + 소요시간`
- `이동시간`: 현재 일정으로 오기 위한 이동칩 시간
- `보정시간`: 이동칩의 여유시간. 일정 사이의 남는 시간이나 강제 조정 차이를 흡수하는 슬롯
- `현재 일정 시작`
- `현재 일정 소요시간`
- `현재 일정 종료`

---

## 기본 계산식

일정 `N`의 시작 가능 기준 시각:

```text
baseArrival(N) = previousEnd(N-1) + travelTime(N)
```

일정 `N`의 일반 시작 시각:

```text
startTime(N) = baseArrival(N) + bufferTime(N)
```

일정 `N`의 종료 시각:

```text
endTime(N) = startTime(N) + duration(N)
```

즉 일반 흐름은 항상 아래와 같다.

```text
이전 일정 종료
+ 이동시간
+ 보정시간
= 현재 일정 시작
+ 소요시간
= 현재 일정 종료
```

---

## 보정시간의 의미

보정시간은 단순 표시값이 아니다.

보정시간은 아래 역할을 한다.

1. 자동계산된 현재 일정 시작과 직접입력된 현재 일정 시작의 차이를 저장한다.
2. 사용자가 시작시간을 뒤로 미뤘을 때 생긴 여유를 저장한다.
3. 잠금 조건이 있어도 타임라인 계산을 끊지 않도록 차이를 흡수한다.
4. 이후 새 일정을 추가할 때 먼저 소비되는 시간 슬롯이다.

즉 보정시간은 `계산 실패`를 뜻하면 안 되고, 오히려 `계산을 계속 이어가기 위한 흡수 버퍼`여야 한다.

---

## 시작시간 고정 규칙

### 원칙

사용자가 일정 시작시간을 직접 바꾸고 `시작시간 잠금`을 켜면, 그 시간이 절대 기준값이 된다.

이 경우 계산 방식은 아래와 같아야 한다.

```text
autoStartTime(N) = baseArrival(N) + currentBuffer(N)
fixedStartTime(N) = userSelectedStart
bufferTime(N) = max(0, fixedStartTime(N) - baseArrival(N))
```

즉:

- 고정 시작시간이 `baseArrival`보다 늦으면 그 차이만큼 보정시간을 늘린다.
- 고정 시작시간이 `baseArrival`와 같으면 보정시간은 `0분`이다.
- 고정 시작시간이 `baseArrival`보다 이르면 시간을 계산 포기하지 말고, 이후 타임라인 재정렬 기준으로 사용해야 한다.

### 강조 표시 규칙

보정시간 색상 강조는 아래 경우에만 켠다.

```text
directStartTime != autoStartTime
```

즉:

- 사용자가 시작시간을 직접 입력하거나 고정해서 자동 시작시각과 차이가 생긴 경우만 강조
- 기본 보정시간 10분 상태는 강조하지 않음
- 단순 기본값 초기화는 강조하지 않음

### 중요한 금지 규칙

아래 동작은 오동작으로 본다.

1. 시작시간을 고정했는데 보정시간을 무조건 `0분`으로 덮어쓰기
2. `고정/잠금으로 계산 불가`라고 띄우고 계산을 멈추기
3. 고정 시작시간을 사용하지 않고 이전 자동 계산값으로 되돌아가기

---

## 소요시간 잠금 규칙

소요시간 잠금은 `duration`을 고정하는 의미다.

```text
duration(N) = fixedDuration
endTime(N) = startTime(N) + fixedDuration
```

소요시간이 잠겨 있어도 시작시간 쪽은 계속 계산해야 한다.

즉:

- 시작시간이 바뀌면 종료시간도 같이 움직인다.
- 계산이 안 맞는 차이는 보정시간으로 보낸다.
- 잠금 때문에 계산을 멈추지 않는다.

---

## 종료시간 잠금 규칙

종료시간 잠금은 `endTime`을 기준값으로 둔다는 뜻이다.

```text
fixedEndTime(N) = userSelectedEnd
duration(N) = fixedEndTime(N) - startTime(N)
```

이때도:

- duration이 바뀔 수는 있어도 타임라인 계산은 계속 이어져야 한다.
- duration이 음수가 되는 경우만 방어하고, 일반적인 차이는 보정시간으로 앞뒤를 조율한다.

---

## 안내 메시지 클릭 보정 규칙

예: `운영 시작 전 방문 (10:00 이후 권장)`

이 경고를 눌렀을 때는 아래처럼 동작해야 한다.

```text
startTime(N) = openTime
bufferTime(N) = max(0, openTime - baseArrival(N))
```

즉:

- 시작시간을 운영 시작 시각으로 맞춘다.
- 그 차이를 이동칩 보정시간에 넣는다.
- `0분 주황 강조` 같은 깨진 상태를 만들면 안 된다.

---

## 일정 추가 규칙

사용자는 보정시간이 넉넉한 구간에 새 일정을 끼워 넣으려는 경우가 많다.

따라서 새 일정 추가 시 규칙은 아래와 같아야 한다.

1. 현재 구간의 보정시간을 먼저 소비한다.
2. 새 일정이 그 보정시간 안에 들어오면 뒤 일정은 밀지 않는다.
3. 보정시간을 초과하는 경우에만 뒤 타임라인을 민다.

보정시간 칩을 그냥 클릭했을 때는 아래 규칙을 따른다.

1. 이후 일정에 시작/소요/종료 잠금이 없으면 `10분` 기본값으로 되돌린다.
2. 이후 일정에 잠금이 있으면 되돌리지 않는다.

즉 보정시간은 실질적인 `삽입 가능 슬롯`이다.

---

## 네비게이션 표시 규칙

네비게이션의 주황색 보정칩은 아래 조건에서 표시한다.

```text
bufferTime >= 30분
```

이 값은 `_isBufferCoordinated` 여부만으로 숨기면 안 된다.

표시 목적은:

- 사용자가 타임라인 상의 여유 구간을 한눈에 보는 것
- 새 일정 삽입 가능 위치를 빠르게 찾는 것

---

## 현재 사용자 요구 기준

현재 사용자가 원하는 계산 개념은 아래와 같다.

```text
이전 일정 종료
+ 이동시간
+ 보정시간
= 현재 일정 시작
+ 소요시간
= 현재 일정 종료
+ 이동시간
+ 보정시간
= 다음 일정 시작
```

그리고 다음 원칙을 반드시 지켜야 한다.

1. 시간은 항상 연속적이어야 한다.
2. 고정/잠금은 계산 불가 사유가 아니라 계산 기준값이다.
3. 일반식으로 안 맞는 차이는 보정시간으로 채운다.
4. 보정시간은 0으로 날려버리지 말고 실제 차이를 보관한다.

## 확인해야 할 대표 버그

### 버그 A

증상:

- 시작시간을 뒤로 미루고 `시작시간 잠금`
- 실제로는 여유시간이 생김
- 그런데 이동칩 보정시간이 `0분`이 되거나 계산 불가가 뜸

정상 동작:

```text
새 고정 시작시간 - baseArrival = 새로운 보정시간
```

### 버그 B

증상:

- 운영 시작 전 경고 클릭
- 시작시간은 바뀌는데 이동칩 보정시간이 비정상적으로 0분/주황강조로 깨짐

정상 동작:

- 운영 시작 시각 기준으로 보정시간 재계산

### 버그 C

증상:

- 보정시간이 충분한데 새 일정 추가 시 뒤 일정이 바로 밀림

정상 동작:

- 현재 보정시간을 먼저 소모한 뒤, 초과분만 뒤로 밀기

## 구현 체크리스트

- [ ] `recalculateSchedule`가 고정/잠금 조건에서도 계산을 포기하지 않는가
- [ ] 시작시간 고정 시 보정시간을 `baseArrival` 기준으로 다시 계산하는가
- [ ] 종료시간 잠금 시에도 타임라인이 연속적으로 이어지는가
- [ ] 안내 메시지 클릭 보정이 이동칩 보정시간으로 연결되는가
- [ ] 보정시간 30분 이상이면 네비게이션에 표시되는가
- [ ] 새 일정 삽입 시 보정시간을 먼저 소비하는가

## 결론

이 프로젝트에서 보정시간은 옵션이 아니다.

보정시간은 `고정/잠금/수동 조정/삽입`을 모두 시간축 위에서 이어 붙이는 핵심 장치다.

따라서 앞으로의 기준은 아래 한 줄로 정리한다.

```text
계산이 안 맞으면 멈추지 말고, 그 차이를 보정시간으로 흡수해서 타임라인을 계속 연결한다.
```
