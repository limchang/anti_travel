# Bug Fix Log

## 2026-03-12

### Bug
- 시간 편집 모달에서 시/분 휠을 드래그하면 모달이 바로 닫혀 정상 조작이 어려웠음.
- 메인 경로 지도는 Day별 점만 단순 연결한 요약선이라 실제 카카오 길찾기 경로가 아니었고, 실패 시 가짜 SVG fallback이 보여 사용자가 실제 지도로 인식하기 어려웠음.

### Cause
- 시간 편집 모달은 `window scroll` 캡처로 닫히도록 되어 있었는데, 모달 내부 휠 리스트 스크롤도 같은 이벤트로 잡아 즉시 닫고 있었음.
- `routeVerify`는 거리/시간만 반환하고 도로 vertex path는 내려주지 않았고, 클라이언트 `RoutePreviewCanvas`도 저장 좌표를 직선으로 이어서만 그리고 있었음.

### Action
- 시간 편집 모달의 scroll close 로직에서 `data-time-modal="true"` 내부 스크롤은 무시하도록 수정.
- `functions/index.js`와 `api/route-verify.js`가 카카오 길찾기 `sections[].roads[].vertexes`를 `path` 배열로 반환하도록 확장.
- `src/App.jsx` 경로 지도 파이프라인을 다시 구성해 Day별 포인트를 먼저 표시하고, 각 연속 포인트 구간은 `routeVerify`로 받은 실제 카카오 도로 path를 선으로 렌더하도록 변경.
- 가짜 섬 SVG fallback은 제거하고, 카카오 지도 렌더 실패 시에는 단순 오류 안내만 남기도록 정리.
- 경로 재로딩 중에도 기존 지도가 있으면 바로 보이게 하고, 로딩 배지만 덧씌우도록 수정.

### Result
- 시간 편집 모달에서 시/분 휠을 드래그해도 바로 닫히지 않음.
- 메인/확대 지도는 카카오 타일 위에 실제 길찾기 path를 그리며, Day 1만 보이던 요약선 구조에서 벗어나 Day별 경로를 실제 도로 기준으로 다시 표시함.

### Bug
- 일정 카드 사이 이동거리 계산이 `주소확인`으로 반복 표시됨.
- 주소와 저장 좌표가 있어도 자동 경로 계산을 누르면 카카오 경로 대신 fallback 경로로 빠지면서 실패하거나 `주소확인`으로 보였음.

### Cause
- 자동 경로 계산이 `api/route-verify` 카카오 검수 호출 실패 시 OSRM/Nominatim fallback으로 내려가고 있었음.
- 정적 배포 환경에서 `/api/route-verify` 고정 호출이 불안정했고, GitHub Pages에서는 이 경로가 보장되지 않았음.
- 서버에 이미 좌표를 넘길 수 있는데도 주소 재지오코딩을 다시 하면서 실패가 늘어났음.

### Action
- 자동 경로 계산은 카카오 `route-verify`만 사용하도록 정리하고 OSRM fallback을 제거.
- 클라이언트가 저장 중인 `geo`, `geoStart`, `geoEnd` 좌표를 `api/route-verify`로 직접 전달하도록 수정.
- `api/route-verify.js`는 전달받은 좌표가 있으면 카카오 로컬 지오코딩을 생략하고 바로 카카오 모빌리티 길찾기를 호출하도록 수정.
- 배포 환경별 엔드포인트 후보를 순차 시도하도록 정리해 `/api/route-verify` 고정 의존을 완화.
- Firebase Functions에 `routeVerify` 공개 함수를 추가하고, 앱에서 `https://asia-northeast3-anti-planer.cloudfunctions.net/routeVerify`도 직접 호출 후보에 포함.

### Result
- 자동 경로 계산 경로가 카카오 API 기준으로 단일화됨.
- 저장 좌표가 있는 일정은 서버 재지오코딩 없이 바로 카카오 길찾기 계산으로 들어감.
- 카카오 경로 호출 자체가 실패하면 `카카오경로실패`로 드러나고, 주소 문제와 분리해서 볼 수 있게 됨.

### Result
- 수동 자동 경로 버튼은 직전 실패와 무관하게 즉시 다시 계산을 시도함.
- 이동칩과 시간 셀 비율이 일정 카드 정보 셀과 더 비슷한 밀도로 정리됨.

### Bug
- 로컬 최신 코드에서도 이동칩이 계속 `주소확인`으로 남는 구간이 있었음.

### Cause
- 경로 계산 주소 판정이 `receipt.address`와 일부 문자열 필드만 보고 있었고, 기존 데이터에 남아 있던 `geo.address`, `geoStart.address`, `geoEnd.address`는 무시하고 있었음.
- 그래서 좌표/주소가 DB에 저장돼 있어도 라벨 단계에서 주소 없음으로 판단하는 케이스가 남아 있었음.

### Action
- `getRouteAddress`가 일반 일정은 `geo.address`, 페리는 `geoStart.address`, `geoEnd.address`까지 fallback 하도록 수정.
- 이동칩 라벨은 `targetItem.distance`가 아직 반영되기 전이라도 `routeCache.distance`가 있으면 바로 표시하도록 보강.

### Result
- 예전 문서 구조나 좌표 우선 저장 상태에서도 이동칩이 저장된 주소/좌표를 사용해 자동 경로 계산과 표시를 이어감.

### Bug
- 자동 경로 전체 계산 중 어떤 일정까지 내려가고 있는지 왼쪽 네비게이션에서 추적하기 어려웠고, 메인 경로 지도도 주소가 충분한데 비어 보이는 경우가 있었음.

### Cause
- 네비게이션은 `calculatingRouteId`를 이동칩 버튼에만 반영하고 사이드바 항목에는 연결하지 않았음.
- 메인/모달 경로 지도는 `day` 단위로 포인트가 2개 이상일 때만 렌더 대상으로 삼아서, 일자별로 1개씩만 주소가 잡힌 경우 전체 주소 수가 충분해도 빈 상태로 떨어졌음.

### Action
- 현재 경로 계산 중인 일정 id를 파생해 왼쪽 네비게이션 항목에 로딩 배지를 붙이고, 자동 계산이 내려갈 때 해당 항목으로 사이드바가 함께 스크롤되도록 연결.
- 경로 지도는 일자별 1개 포인트도 렌더 대상으로 포함하고, 총 포인트 수가 충분한데 좌표 확인이 아직 안 된 상태는 별도 안내 문구로 분리.

### Result
- 자동 경로 계산 진행 상황을 네비게이션에서 바로 따라갈 수 있게 됐고, 메인 지도는 일자별 주소 포인트가 분산된 일정에서도 실제 카카오 지도/마커가 더 안정적으로 보이게 됨.

## 2026-03-12

### Bug
- 로컬에서 메인 경로 지도가 계속 `주소 좌표` 안내만 띄우고 실제 지도를 못 그리는 경우가 있었음.

### Cause
- 브라우저 geocode가 카카오 REST / Nominatim fallback에 기대고 있었고, 이미 로드된 카카오 지도 SDK에는 `services` 라이브러리가 없어 JS SDK 지오코더를 활용하지 못했음.

### Action
- 카카오 지도 SDK를 `libraries=services`로 로드하고, 경로 미리보기 geocode는 JS SDK `Geocoder` / `Places`를 최우선으로 사용하도록 변경.
- 안내 문구도 `좌표` 대신 `지도 위치` 기준으로 정리.

### Result
- 로컬에서도 카카오 JS SDK 기준으로 주소를 더 안정적으로 지도 위치로 변환해 메인 경로 지도가 표시되도록 보강됨.
