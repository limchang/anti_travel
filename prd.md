# PRD — Anti Planer

## 1. 문서 목적
- 이 문서는 현재 `anti_planer` 프로젝트의 실제 구현 상태를 기준으로 제품 목표, 사용자 문제, 핵심 기능, 데이터 구조, 운영 제약을 정리한 현행 PRD다.
- 신규 기능 설계, 리팩터링 우선순위 결정, AI/백엔드 연동 범위 확인 시 기준 문서로 사용한다.

## 2. 제품 한줄 정의
`Anti Planer`는 여행 일정을 "고정된 표"가 아니라 "실시간으로 조정 가능한 운영 보드"처럼 다루게 해주는 여행 일정 편집 앱이다.

## 3. 문제 정의
기존 여행 계획 방식은 아래 문제를 반복한다.

- 장소 추가와 재배치가 번거롭다.
- 영업시간, 이동시간, 버퍼시간을 반영하기 어렵다.
- 지도에서 복사한 정보를 일정에 다시 입력하는 과정이 길다.
- 여행 중 예산, 대안 장소, 공유 권한을 한 화면에서 관리하기 어렵다.

본 제품은 이 문제를 줄이기 위해 일정표, 장소 보관함, AI 자동입력, 경로 검증, 예산 추적을 하나의 흐름으로 묶는다.

## 4. 목표
### 제품 목표
- 모바일 중심 여행 일정 편집 경험 제공
- 일정 추가, 이동, 삭제, 대안 추가를 드래그 기반으로 단순화
- 장소 정보 입력 시간을 AI와 링크/스크래핑으로 단축
- 실제 이동시간과 운영정보를 반영한 더 현실적인 일정 관리 지원
- 개인 계획, 공유 링크, 읽기 전용/편집 권한까지 한 제품에서 처리

### 비즈니스/운영 목표
- Firebase 기반으로 사용자별 일정 저장
- 서버 API를 통해 외부 AI 키와 분석 기능을 안전하게 중계
- Vercel API와 Firebase Functions를 병행할 수 있는 배포 구조 유지

## 5. 대상 사용자
### 1차 타깃
- 국내 여행 일정을 직접 세밀하게 짜는 사용자
- 제주 등 다일차 여행에서 식당, 카페, 관광, 숙소를 시간 단위로 조정하는 사용자
- 네이버 지도/텍스트를 자주 복사해 일정에 붙여넣는 사용자

### 2차 타깃
- 동행자와 링크로 일정을 공유하려는 사용자
- 여행 중 "플랜 B" 후보 장소를 함께 관리하려는 사용자

## 6. 핵심 사용자 가치
- 일정이 바뀌어도 전체를 다시 쓰지 않고 바로 재조정할 수 있다.
- 장소 정보를 수동 입력하지 않아도 된다.
- 이동시간, 버퍼시간, 영업시간, 예산을 동시에 볼 수 있다.
- 본 일정과 후보 장소를 한 화면에서 운영할 수 있다.

## 7. 핵심 사용자 시나리오
### 시나리오 A. 새 여행 시작
1. 사용자는 로그인하거나 게스트로 진입한다.
2. 여행 지역, 시작일, 종료일, 예산을 설정한다.
3. 기본 일정 혹은 빈 일정이 생성된다.

### 시나리오 B. 장소 추가 및 일정 편집
1. 사용자는 `내 장소` 라이브러리에 장소를 저장한다.
2. 저장한 장소를 타임라인에 드래그해 일정에 배치한다.
3. 시간, 소요시간, 버퍼시간, 메모, 메뉴, 영업정보를 수정한다.
4. 필요 시 대안 장소(Plan B)를 본 일정 카드에 연결한다.

### 시나리오 C. AI 스마트 채우기
1. 사용자는 네이버 지도 텍스트 또는 링크/이미지를 복사한다.
2. 앱은 Gemini, Groq, 스크래퍼, 검색 fallback을 이용해 장소명/주소/영업정보/메뉴를 추출한다.
3. 사용자는 결과를 검토 후 수정한다.
4. 수정 내용은 AI 학습용 교정 데이터로 저장될 수 있다.

### 시나리오 D. 이동시간 검증
1. 사용자는 두 일정 사이 자동경로 버튼을 누른다.
2. 앱은 서버에서 카카오 길찾기를 우선 호출하고 필요 시 보정/대체 경로를 사용한다.
3. 계산된 거리와 시간이 이동 칩에 반영된다.

### 시나리오 E. 공유
1. 사용자는 일정 공개 범위와 편집 권한을 설정한다.
2. 공유 링크를 복사해 다른 사용자에게 전달한다.
3. 링크 진입자는 읽기 전용 또는 편집 권한으로 일정에 접근한다.

## 8. 기능 요구사항
### 8.1 인증 및 사용자 세션
- Google 로그인 지원
- 게스트 모드 지원
- 사용자별 Firestore 일정 저장
- 인증 사용자는 복수 일정 문서를 가질 수 있어야 함

### 8.2 여행 메타 정보
- 여행 지역, 시작일, 종료일 설정
- 시작일/종료일 기준 여행 일수 계산
- 플랜 제목, 플랜 코드 생성 및 표시
- 총 예산(`maxBudget`) 설정

### 8.3 일정 타임라인
- 일자별 일정 목록 관리
- 각 일정 아이템은 시간, 소요시간, 버퍼시간, 타입, 메모, 주소, 영업정보, 영수증/메뉴 정보를 가질 수 있어야 함
- 시간 잠금, 종료시간 잠금, 소요시간 잠금 상태를 지원
- Undo 토스트 기반 최근 변경 되돌리기 제공
- 일정 간 이동 칩에 이동시간/거리/버퍼 조절 UI 제공

### 8.4 장소 라이브러리
- `내 장소` 목록 저장
- 라이브러리 → 일정 드래그 추가
- 일정 → 라이브러리 이동 또는 복제
- 타입 태그, 메모, 주소, 메뉴, 영업정보를 유지

### 8.5 대안 장소(Plan B)
- 일정 카드마다 대안 장소 목록을 붙일 수 있어야 함
- 라이브러리 장소 또는 다른 일정 아이템을 대안으로 연결 가능해야 함
- 대안을 본 일정으로 승격하거나 라이브러리로 이동할 수 있어야 함

### 8.6 AI 스마트 채우기
- 주소 전용, 영업정보 전용, 메뉴 전용, 전체 입력 모드 지원
- 텍스트 기반 분석 우선
- 네이버 링크 분석 지원
- 이미지 포함 입력 지원
- 링크만 단독 입력했을 때 사용자 가이드/방어 처리 필요
- 사용자 수정 결과를 AI 학습 피드백으로 저장할 수 있어야 함

### 8.7 장소 정보 수집
- Gemini 링크 분석 API
- Groq 분석 API
- Naver 페이지 스크래핑 API
- 장소명 기반 주소 탐색 fallback
- 네이버 `place id` 저장 및 재사용 지원

### 8.8 경로 계산
- 주소 기반 좌표 변환
- 카카오 길찾기 우선
- OSRM fallback 지원
- 직선거리/도로거리 기반 최소 시간 검증 로직 제공
- 계산 결과 캐싱 및 재시도 쿨다운 제공

### 8.9 예산 관리
- 일정별 메뉴/영수증 항목 가격 저장
- 총 사용 예산 합계 계산
- 남은 예산 및 사용률 표시

### 8.10 공유 및 협업
- `private`, `link`, `public` 공개 범위 지원
- `viewer`, `editor` 권한 지원
- 공유 진입 시 읽기 전용 상태 제어

### 8.11 푸시 및 운영 정보
- FCM 권한 요청 및 토큰 발급
- 포그라운드 메시지 수신 처리
- 앱 버전 및 업데이트 로그 표시
- 마지막 배포/업데이트 인지 보조 UI 제공

## 9. 데이터 요구사항
### 핵심 문서 구조
```js
{
  days: [
    {
      day: 1,
      date: "2026-03-26",
      plan: [/* timeline items */]
    }
  ],
  places: [/* place library */],
  maxBudget: 1500000,
  tripRegion: "제주",
  tripStartDate: "2026-03-26",
  tripEndDate: "2026-03-28",
  planTitle: "제주 여행",
  planCode: "JEJU-20260326-XXXX",
  share: {
    visibility: "private",
    permission: "viewer"
  }
}
```

### 저장 위치
- 로그인 사용자: `users/{uid}/itinerary/{planId}`
- 공유/레거시 fallback: `itinerary/main`
- AI 설정/학습 관련 메타: `users/{uid}/private/*`, `meta/smartFillGuide`

## 10. 비기능 요구사항
- 모바일 화면에서 사용 가능해야 함
- 드래그 앤 드롭이 핵심 인터랙션으로 자연스럽게 동작해야 함
- 저장은 자동 저장이어야 하며 사용자가 명시적으로 저장 버튼을 누르지 않아도 됨
- 서버 API는 CORS 및 OPTIONS 요청을 처리해야 함
- 외부 AI 키는 브라우저 평문 저장보다 서버 암호화 저장을 우선해야 함
- 실패 시 fallback 경로를 제공해야 함

## 11. 성공 지표
- 장소 추가부터 일정 반영까지의 단계 수 감소
- AI 스마트 채우기 후 사용자의 수동 수정 횟수 감소
- 자동경로 계산 성공률 향상
- 공유 링크 진입 후 이탈률 감소
- 일정당 라이브러리/대안 장소 활용 비율 증가

## 12. 현재 확인된 제약 및 리스크
- 프론트엔드 핵심 로직이 `src/App.jsx` 단일 대형 파일에 집중되어 유지보수 난도가 높다.
- AI 정확도는 입력 품질, 외부 API 상태, 프롬프트 설계에 크게 의존한다.
- 네이버 스크래핑은 외부 DOM 구조 변경에 취약하다.
- 경로 계산은 주소 품질이 낮으면 정확도가 떨어질 수 있다.
- Firebase Admin 환경변수나 외부 AI 키가 누락되면 일부 기능이 제한된다.

3. 다인 협업 충돌 처리 정책 정의
4. 일정 추천/자동 최적화 고도화
5. 테스트 코드 및 회귀 검증 체계 확립
